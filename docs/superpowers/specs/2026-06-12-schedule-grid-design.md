# Спека: рабочий грид смен (Этап 1)

- **Дата:** 2026-06-12
- **Статус:** утверждена основателем (брейнсторм-сессия)
- **Связано:** ROADMAP.md → Этап 1; дизайн-источник: `components/marketing/grid-preview.tsx`

## Цель

Превратить заглушку `/schedule` в рабочий грид смен на реальных данных Supabase. Дизайн и UX **не изобретаем** — они зафиксированы демо-гридом на лендинге; задача — воспроизвести их на производственном фундаменте.

## Ключевые решения (утверждены)

1. **Подход B: новый продуктовый грид по дизайну демо.** Демо (7107 строк, маркетинговый код с фейковыми типами и без виртуализации) не форкается и не рефакторится — оно остаётся дизайн-спекой и живёт на лендинге. Продуктовый грид — новая модульная реализация в `components/schedule/*` на утверждённом стеке: TanStack Table v8 + TanStack Virtual, TanStack Query, optimistic upsert. Визуальный слой (токены, классы ячеек, бейджи, режимы) переносится из демо; при сборке каждого блока — визуальная сверка с демо бок о бок.
2. **Скоуп: полный паритет с демо**, включая 3 режима ячеек (compact / detail / extended), типы смен с временем (утро / вечер / ночь) и вкладку «Сотрудники» (карточки / список).
3. **Фичи-сироты скрываем до своих этапов:** проекты с командами (вне роадмапа), таймер смены (тайм-трекинг вне MVP), Telegram-кнопки (этап 5). В продуктовом гриде эти элементы не рендерятся.
4. **Первый запуск: быстрый старт.** Пустое состояние с пошаговым CTA: создай отдел → добавь сотрудников → кликай ячейки. CSV-импорт — отдельный мини-этап после ядра (лендинг его обещает, но в этап 1 не входит).

## Данные и поток

### Загрузка месяца

`app/(app)/(shell)/schedule/page.tsx` (Server Component) параллельно выполняет 4 запроса под RLS:

- `employees` активные (`deleted_at IS NULL`), отсортированы по `dept_id, sort_order`
- `departments` организации
- `status_types`: системные (`org_id IS NULL`) + кастомные организации
- `schedule_entries` за месяц: `org_id = X AND entry_date BETWEEN first AND last` (главный индекс `(org_id, entry_date)`)

Клиент: TanStack Query, ключ `['schedule', orgId, 'YYYY-MM']`, начальные данные гидрируются с сервера. Записи маппятся в `Map<employeeId, Map<dateISO, Entry>>` — O(1) доступ из каждой ячейки.

### Редактирование ячейки

Клик по ячейке → попап-редактор: выбор статуса, пресеты смен, время, «очистить».

- **Пресеты «утро / вечер / ночь»** — быстрые кнопки, проставляющие `start_time`/`end_time` при статусе «работает». Дефолтное окно статуса — из `status_types.start_time/end_time`; точечный override пишется в `schedule_entries.start_time/end_time`.
- **Optimistic update:** мутация мгновенно обновляет Query-кэш → server action `upsertEntryAction` (уникальность `(employee_id, entry_date)` enforced БД) → при ошибке откат кэша + тост.
- **Конфликты:** last-write-wins; `created_by`/`updated_by` штампуются существующим триггером `set_schedule_entry_actor`.

### Миграция (единственная)

`employees.employment_kind text check in ('staff','trainee') default 'staff'` — под бейдж «стажёр / штатный» из демо. Цветные роли демо маппятся на существующий `employees.position` (свободный текст, нейтральное отображение). Других изменений схемы нет.

### Время

«Сегодня» (подсветка колонки) вычисляется в таймзоне организации (`organizations.timezone`), не браузера. `entry_date` — тип `date`, без TZ-конверсий.

- Виртуализация строк (TanStack Virtual): 300 сотрудников без лагов; 31 колонка дней рендерится целиком.
- Группировка по отделам — сворачиваемые секции (как в демо).
- Drag-and-drop порядка сотрудников — `@dnd-kit` (новая зависимость).
- CRUD сотрудников/отделов: модалки из грида («+ Сотрудник») и та же форма на вкладке «Сотрудники».

## Поведение и права

- **Роли:** owner/admin — всё; manager — правка только своих отделов (`visible_department_ids()`), остальное по RLS; viewer — read-only. UI прячет недоступные действия, финальная защита — RLS + проверки в server actions (`assertCan`).
- **Read-only при истёкшем триале:** `isReadOnly` из shell-контекста — редактор ячеек не открывается, баннер уже существует.
- **Покрытие:** строка под шапкой: на день и отдел — число `counts_as_present` против `alert_configs.min_present`; дефицит подсвечивает колонку, бейдж «не хватает N». Пороги настраиваются per-отдел.
- **Фильтры:** отдел, поиск по имени, режим — все в URL search params (ссылка шарится).
- **Лимиты плана:** финальная проверка в `createEmployeeAction` через `checkPlanLimit`; UI заранее показывает «N из лимита».

### Edge cases

- Ночная смена через полночь (`end < start`) — валидна, бейдж «ночь».
- Сотрудник без отдела — виден owner/admin (RLS уже учитывает), группа «Без отдела».
- Удаление сотрудника — мягкое (`deleted_at`); записи остаются в БД, строка исчезает из грида.
- Перенос сотрудника между отделами не трогает его записи.
- Месяцы 28–31 день; неделя с понедельника; выходные подсвечены (как в демо).

## Тестирование (verification-before-completion)

1. **Юниты (vitest):** маппинг месяца в Map, расчёт покрытия, Zod-схемы, «сегодня в TZ организации».
2. **БД-интеграционные (set_config + rollback):** менеджер не пишет в чужой отдел; viewer не пишет; кросс-org записи невозможны; лимиты плана.
3. **E2E в превью:** quick start → отдел → 3 сотрудника → смены → покрытие → drag-and-drop → фильтры → режимы. Визуальная сверка с демо.
4. **Перф:** сид 300 сотрудников × месяц; скролл плавный, правка ячейки мгновенная.
5. **После этапа:** tsc + vitest + build + 🔒 security review (блок в ROADMAP: server actions, RLS, лимиты).

## Definition of Done

Организация с 30+ сотрудниками ведёт реальный месяц; смена статуса мгновенна; покрытие подсвечивается; три режима и вкладка «Сотрудники» соответствуют демо; тесты RLS зелёные; роли работают как описано; перф-чек на 300 сотрудников пройден.
