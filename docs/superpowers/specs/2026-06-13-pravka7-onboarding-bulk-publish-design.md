# Правка 7 — онбординг, порядок меню, bulk-добавление, чистка сайдбара, публикация графика

- **Дата:** 2026-06-13
- **Статус:** дизайн утверждён основателем (чат, «да»)
- **Решения основателя:** публикация по месяцам; «Уведомить» пишет выбор в БД, реальная рассылка позже; админ сохраняет права на правку/публикацию.

## 1. Онбординг — экран настройки компании

Референс — 7shifts «3 minutes until you're in the app» (split-layout).

- **Лейаут:** слева фирменная панель (крем `--background`, знак Smengo, честные value-props
  «14 дней бесплатно · отмена в любой момент · карта не нужна» — БЕЗ фейковых тестимониалов),
  справа форма на белой/кремовой подложке. Мобильный: панель скрывается, форма во всю ширину.
- **Поля:** название компании (есть), **кол-во сотрудников** — чипы `1–15 / 16–50 / 51–150 / 150+`
  (опционально), **аватар компании** (опционально: выбор файла + круглое превью, клиентское сжатие
  как у фото сотрудников), для OAuth-флоу остаются «откуда узнали» + чекбокс условий.
  Поведение префилла из user_metadata не меняется.
- **БД:** `organizations.team_size text` (+ колонка в grant update). RPC `create_organization`
  НЕ меняем: после успешного создания org сервер-экшен дописывает `team_size` обычным UPDATE
  (RLS org_update, владелец = создатель) и грузит лого через логику `uploadOrgLogoAction`
  (бакет `org-logos` существует, политики owner-only записи проходят).
- **i18n:** ключи ru/uk/en в `auth.onboarding.*`.

## 2. Порядок меню и редирект после логина

- Сайдбар: **Дашборд → График → Сотрудники**.
- После логина — на `/schedule`: правки в `lib/supabase/middleware.ts`

- Редирект «без организации → /onboarding» не трогаем.

## 3. Массовое добавление сотрудников (таблица)

Референс — 7shifts «Add your employees».

- **Модалка-таблица**, колонки: `Имя* · Отдел (select) · Должность · Email · Телефон`.
  3 пустых строки на старте; при заполнении имени в последней строке автоматически
  добавляется новая; строку можно удалить. Кнопка «Добавить N сотрудников» активна,
  когда есть ≥1 строка с непустым именем.
- **Server action** `bulkCreateEmployeesAction(rows[])`: Zod-массив (имя required, email/телефон
  опциональны с форматной проверкой), гейт `crud_employees`, **лимит плана проверяется на сервере
  на всю пачку** (текущее число + N ≤ лимит), вставка одним insert-батчем с org_id.
  Částичных вставок нет: превышение лимита или ошибка валидации любой строки = отказ всей пачки
  с указанием строки.
- **Входные точки:** кнопка «+ Сотрудник» тулбара и ghost-плашка пустого отдела (отдел предвыбран).
  Существующая одиночная модалка остаётся только для редактирования.

## 5–6. Публикация графика

- **Таблица `schedule_publications`:** `id, org_id, month date (первое число), published_by uuid,
  notify boolean, published_at timestamptz default now()`. RLS: select — члены организации;
  insert — owner/admin/manager (`auth_has_role`). Индекс `(org_id, month, published_at desc)`.
- **Детекция правок — `schedule_change_marks (org_id, month, last_change_at)`:**
  AFTER INSERT/UPDATE/DELETE-триггер на `schedule_entries` делает upsert метки месяца.
  Вариант `max(updated_at)` отвергнут: не ловит удаления записей.
  RLS: select — члены организации; пишет только триггер (SECURITY DEFINER-функция).
- **Кнопка «Опубликовать график»** в тулбаре /schedule: видна только при
  `can(role,'edit_schedule')` И когда месяц «грязный» (`last_change_at > последняя публикация`,
  локально появляется сразу после первой мутации). Акцентный стиль, иконка Send.
- **Диалог публикации** (как 7shifts): радио «Уведомить всех» / «Не уведомлять» → кнопка
  «Опубликовать». `publishScheduleAction({month, notify})`: гейт edit_schedule + insert.
  После успеха — тост, кнопка гаснет. Реальная рассылка писем — отдельная правка
  (выбор notify уже сохранён в БД).
- **fetchMonthData** дополняется `lastChangeAt` и `lastPublishedAt` месяца.

## Не входит

Реальная отправка уведомлений; изменение ролей (admin сохраняет права); миграция RPC
`create_organization`; история публикаций в UI.

## Тестирование

tsc/eslint/vitest; юнит-тесты на Zod-схему bulk-строк и «грязную» логику (dirty-вычисление);
RLS-проверка publications/marks (insert чужой org — отказ); живой E2E в превью: онбординг
(новая организация создаётся и удаляется), bulk-добавление 3 сотрудников, публикация
(кнопка появляется после правки, гаснет после публикации), откат тестовых данных.
